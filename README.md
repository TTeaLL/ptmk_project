# Система учёта заявок сотрудников

Тестовое задание: консольное приложение для управления заявками с этапами «Новая → В работе → Выполнена».  
Реализовано на **Python** с использованием **SQLAlchemy** и **SQLite**.

## Бизнес-процесс

Заявка проходит три этапа:

1. **Новая** (`new`) – только создана, ожидает назначения исполнителя.
2. **В работе** (`in_progress`) – исполнитель приступил к выполнению.
3. **Выполнена** (`done`) – работа завершена.

Переход осуществляется последовательно: `new` → `in_progress` → `done`.

---

## Бизнес-правила

- Из статуса `new` можно перейти только в `in_progress`.
- Из `in_progress` можно перейти только в `done`.
- Из `done` переходы запрещены.
- Изменение исполнителя разрешено только для заявок в статусах `new` или `in_progress` (нельзя менять у выполненных).

Все переходы проверяются на уровне сервиса.

## Структура базы данных

### Сущность `employees`
- `id` – первичный ключ
- `full_name` – ФИО
- `department` – подразделение
- `position` – должность

### Сущность `requests`
- `id` – первичный ключ
- `number` – уникальный номер заявки
- `created_date` – дата создания
- `author_id` – внешний ключ к `employees` (автор)
- `executor_id` – внешний ключ к `employees` (исполнитель)
- `description` – описание
- `deadline` – срок выполнения
- `status` – статус (`new`, `in_progress`, `done`)

**Обоснование:**  
Данные нормализованы до 3-й нормальной формы – информация о сотрудниках не дублируется, связи реализованы через внешние ключи. Это обеспечивает целостность и упрощает расширение.

## Запуск и установка

1. Убедитесь, что установлен Python 3.8+.
2. Клонируйте репозиторий и перейдите в папку проекта.
3. Установите зависимости:
   ```bash
   pip install -r requirements.txt

   python main.py
   ```

Важно: при первом запуске автоматически генерируются 1000 сотрудников и 1 000 000 заявок – это может занять 5–15 минут в зависимости от мощности компьютера. При последующих запусках данные уже будут существовать, и программа будет работать быстро.

Работа с приложением
После запуска выполняются следующие действия:

Демонстрация бизнес-правил – попытка перевести заявку из статуса done в in_progress (запрещено), из in_progress в done (разрешено), из new в done (запрещено). Вывод показывает, что правила работают.

Фильтрация – выводятся просроченные заявки в статусе «В работе» для первого исполнителя.

Отчётность – выводятся:

количество заявок по каждому статусу;

общее количество просроченных заявок;

количество выполненных заявок по каждому исполнителю (первые 5).

Замер производительности – выполняется запрос «просроченные заявки конкретного исполнителя в статусе «В работе», отсортированные по сроку выполнения» до и после создания индекса. Результаты выводятся в консоль.

Оптимизация производительности
Исходный запрос (без индекса)

   ```bash
   SELECT * FROM requests
   WHERE executor_id = ?
     AND status = 'in_progress'
     AND deadline < NOW()
   ORDER BY deadline;
   ```

При отсутствии индексов СУБД выполняет полное сканирование таблицы (1 млн записей).

Оптимизация
Создан составной индекс:

   ```bash
	CREATE INDEX idx_executor_status_deadline
	ON requests (executor_id, status, deadline);
   ```

Почему это ускоряет:

Индекс покрывает все условия WHERE и поле сортировки ORDER BY.

Поиск по executor_id и status выполняется за логарифмическое время, а deadline уже отсортирован внутри индекса.

СУБД обращается только к отфильтрованным строкам (без сканирования всей таблицы).

Результаты замеров
Этап	Время (сек)	Количество найденных
Без индекса	0.0070	276
С составным индексом	0.0030	276
Ускорение: ~2.34×

Дополнительные возможности
При необходимости систему можно расширить:

Согласование руководителем – добавить таблицу approvals и новые статусы.

Несколько исполнителей – реализовать связь многие-ко-многим через таблицу request_executors.

История изменений – хранить логи переходов статусов.

Типы заявок – задавать дедлайн в зависимости от типа.

Права доступа – ввести роли (администратор, руководитель, сотрудник) с проверками на уровне сервиса.

Архитектура (модели + сервисный слой) позволяет вносить такие изменения без переписывания основного кода.
